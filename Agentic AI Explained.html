Levels of AI Agents
  • - All AI agents work off a generative model (e.g., LLM, SLM) which has pre-trained knowledge.
Level 1: Basic Model Interaction
  • - Uses only pre-trained knowledge from its training corpus.
  • - Limited by a knowledge cutoff date.
  • - Cannot access private or organizational data.

Level 2: Retrieval Augmented Generation (RAG)
  • - Augments the model with an additional body of knowledge.
  • - Enables access to private data sources like SharePoint, Dataverse, or uploaded files.
Mechanism: Semantic Indexing
  • - Uses an embedding model to create a vector for each piece of data.
  • - The vector represents the semantic meaning, not just keywords.
  • - When a query is made, its vector is compared to the data vectors to find the 'nearest neighbors' (most relevant information).


Level 3: Task/Action Performance
  • - Agent can perform actions by calling external tools.
  • - Examples: Calling APIs, hooking into serverless capabilities.
  • - This makes it a 'task agent'.

Commonality: Human-in-the-Loop
  • - These initial levels typically involve a human as a constant part of the interaction loop.


Agentic AI: The Next Level
  • - An evolution where AI agents can act autonomously to achieve complex, long-running goals.
  • - All agentic agents are AI agents, but not all AI agents are agentic.
Core Characteristics
Dynamic & Complex Planning
  • - Dynamically plans how to achieve a goal, rather than following a predefined workflow.
  • - Uses reasoning to ascertain the optimal plan based on its core instructions.

Autonomous Operation
  • - Triggered by an event, not necessarily direct human interaction.
  • - Can be proactive, suggesting and proposing actions.
Example Triggers
  • - Keywords in an email.
  • - A new record written to a spreadsheet.
  • - Speech recognition.
  • - A scheduled recurrence (e.g., cron job).


Human Interaction (Optional)
  • - May have checkpoints for human approval, especially for risky or costly actions (e.g., spending money).
  • - The human's involvement in the overall loop can vary greatly.

Improvement Over Time (Debated)
  • - Often stated as an attribute, but not common in practice today.
  • - The underlying model is stateless and read-only; it has no memory.
  • - 'Memory' is simulated by storing state externally (e.g., in a Cosmos DB) and retrieving it for new tasks.
  • - True model improvement would require fine-tuning with reinforcement learning signals, which is currently rare.



Multi-Agent Systems
  • - A common and recommended approach for building robust agentic AI solutions.
  • - Avoids creating a single, massive agent which is more prone to hallucination.
Expert Agents
  • - Strategy involves creating many agents, each an expert in a specific, focused task.
  • - This narrow scope makes them less prone to hallucination.

Collaboration & Orchestration
  • - Agents can call other agents to solve more complex problems.
  • - An orchestrator agent can manage the primary mission and delegate tasks to specialized agents.

Validation Patterns
  • - Use multiple agents for checks and balances to increase confidence in autonomous outputs.
  • - Common pattern: Author-Critic or Maker-Checker.


Implementation Examples
Low-Code: Co-pilot Studio
  • - A low-code platform for building AI agents.
Key Features for Agentic AI
  • - Generative Orchestration: Must be enabled to allow the agent to dynamically plan how to respond to users and events.
  • - Triggers: Enables autonomous operation by defining what events should activate the agent.

Configuration Steps
  • - Provide a clear name and description for inter-agent understanding.
  • - Write detailed instructions/directives for the agent's goal.
  • - Add Knowledge Sources (e.g., files, SharePoint, Azure AI Search).
  • - Add Actions (e.g., APIs to send an email).
  • - Configure Triggers (e.g., 'When a new email arrives').

Practical Tip
  • - When using the 'send email' action, the video recommends using version V2, as V3 is reported to not work as well.


Pro-Code: Semantic Kernel
  • - Microsoft's enterprise-ready, open-source orchestration engine for AI development.
  • - Available for .NET, Java, and Python.
Core Concepts
  • - Uses 'connectors' to abstract interactions with various components.
  • - Components include AI providers (models), memory (vector stores), and functions (tools).

Agent Types
  • - Chat Completion Agent: A basic agent for model interaction. Requires manual plumbing for state and tooling.
  • - Assistant Agent: A more advanced type that includes built-in state storage and tooling.

Multi-Agent Implementation
  • - Individual expert agents are defined with specific instructions (system prompts).
  • - A 'Group Chat' is created to allow these agents to work together and converge on a solution.
Group Chat Controls
  • - Termination Strategy: A function must be defined to determine when the goal is met. Example: Terminate when a critic agent's response includes the word 'approved'.
  • - Maximum Iterations: A limit must be set to prevent infinite loops. Example: Stop after 10 iterations.




Operational Considerations & Guardrails
  • - Agentic AI still requires standard operational practices and operates within defined boundaries.
Testing & Evaluation
  • - Perform unit testing on individual components.
  • - Evaluate agent performance with various inputs, often using another LLM as a 'judge'.

Monitoring & Tracing
  • - Implement full logging and tracing (e.g., using Azure AI Foundry's tracer decorator).
  • - Semantic Kernel emits Open Telemetry data, which can be sent to services like App Insights.

Security & Permissions
  • - Agents operate within the guardrails and permissions you provide.
  • - They run under a specific identity context (e.g., a user account or service principal).
  • - Existing controls like Data Loss Prevention (DLP) policies still apply to actions like sending emails.
  • - Highlights the critical need for proper data discovery, classification, and protection within an organization.
